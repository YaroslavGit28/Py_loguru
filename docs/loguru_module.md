## Документация по Loguru 

Цель этого текста — не просто перечислить возможности Loguru,
а объяснить ключевые идеи **простым языком**, так чтобы
разобрался человек, который впервые настраивает логирование в Python.

---

### 1. Что такое Loguru и зачем он нужен

Когда программа работает, нам важно понимать:

- что она делает прямо сейчас;
- где произошла ошибка и почему;
- какие данные были на входе;
- в каком месте кода это случилось.

Для этого используют **логи** (log messages).

В Python есть стандартный модуль `logging`, но новичкам он часто кажется громоздким:
нужно настраивать логгеры, хендлеры, форматтеры и уровни, а “быстрый старт” получается не таким уж быстрым.

**Loguru** решает эту боль:

- даёт готовый `logger`, которым можно пользоваться сразу;
- позволяет настраивать вывод в консоль/файлы/JSON одним методом `logger.add(...)`;
- помогает красиво логировать исключения;
- поддерживает контекст (`extra`) для структурированных логов.

Установка:

```bash
pip install loguru
```

---

### 2. Главный объект: `logger`

В Loguru обычно достаточно:

```python
from loguru import logger

logger.info("Привет! Это INFO‑сообщение.")
```

Логгер “знает”:

- уровень записи (INFO, DEBUG, …),
- текст сообщения,
- место в коде (файл/функция/строка),
- время,
- дополнительные поля контекста (`extra`).

---

### 3. Уровни логирования (DEBUG/INFO/…)

Уровень — это “важность” сообщения.
Обычно применяют:

- `DEBUG` — много деталей, полезно при разработке;
- `INFO` — нормальный рабочий поток;
- `WARNING` — подозрительная ситуация, но продолжаем;
- `ERROR` — ошибка: операция не выполнена;
- `CRITICAL` — очень серьёзная проблема.

Пример:

```python
logger.debug("Детали для отладки: x={}", x)
logger.info("Запрос обработан")
logger.warning("Ответ медленный: {} ms", ms)
logger.error("Не удалось открыть файл: {}", path)
logger.critical("Сервис недоступен")
```

Важно: какие уровни вы увидите — зависит от настройки sink’ов (см. дальше).

---

### 4. Sink’и (куда пишутся логи) и почему их может быть несколько

Sink — это “приёмник” логов: куда Loguru отправляет записи.
Типовые sink’и:

- консоль;
- файл;
- функция (callback);
- поток/объект (например, похожий на файл).

Почему несколько sink’ов — это нормально:

- в консоль — красиво и удобно для человека;
- в файл — чтобы хранить историю;
- в JSON — чтобы анализировать программно;
- отдельный файл — только для важной подсистемы (например, “БД”).

---

### 5. `logger.remove()` и `logger.add(...)`: базовая конфигурация

Частая практическая схема:

1) убрать дефолтные sink’и:

```python
logger.remove()
```

2) добавить свои sink’и:

```python
logger.add("logs/app.log", level="INFO")
```

Это помогает избежать “дублирования” сообщений и держать конфигурацию в одном месте.

В проекте (`guide_loguru.py`) это разделено на две функции:

- `configure_console()` — консольный вывод;
- `configure_files()` — файловые sink’и.

---

### 6. Форматирование: плейсхолдеры и стиль сообщений

В Loguru формат задаётся строкой `format=...`.
Пример (упрощённо):

```python
logger.add(
    sink,
    format="{time:YYYY-MM-DD HH:mm:ss} | {level} | {message} | {extra}"
)
```

Плейсхолдеры:

- `{time:...}` — время с форматированием;
- `{level}` — уровень;
- `{message}` — текст сообщения;
- `{extra}` — словарь контекста.

Отдельно про форматирование сообщений:

```python
logger.info("User id = {}", 42)
```

Это **безопаснее и быстрее**, чем f‑строки “в лоб”, потому что форматирование контролируется логгером.

---

### 7. Кастомные уровни: `logger.level()` и `logger.log()`

Иногда хочется уровень “между” стандартными (например, NOTICE).
Loguru позволяет объявить свой:

```python
logger.level("NOTICE", no=25, color="<yellow>", icon="!")
logger.log("NOTICE", "Сообщение уровня NOTICE")
```

В `guide_loguru.py` это показано в разделе про кастомный уровень.

---

### 8. “Ленивые вычисления”: `logger.opt(lazy=True)`

Проблема:

```python
logger.debug("Сложное значение: {}", expensive())
```

`expensive()` выполнится **сразу**, даже если сообщение будет отфильтровано уровнем.

Решение:

```python
logger.opt(lazy=True).debug("Сложное значение: {}", expensive)
```

Теперь `expensive` вызовется только если запись действительно будет залогирована.

В `guide_loguru.py` это демонстрируется так, чтобы было видно, сколько раз вызывается “дорогая” функция.

---

### 9. Контекст (`extra`): `bind()` и `contextualize()`

Контекст — это дополнительные поля, которые приклеиваются к записи.
Пример: `request_id`, `user_id`, `topic`.

`bind()` создаёт логгер с постоянными полями:

```python
log = logger.bind(user_id=42)
log.info("Запрос обработан")
```

`contextualize()` добавляет поля внутри блока:

```python
with logger.contextualize(request_id="req_123"):
    logger.info("Внутри блока request_id будет в extra")
```

Плюс: это удобно для фильтрации и аналитики.

---

### 10. Фильтрация: `filter=...`

Фильтр позволяет решать, какие записи попадут в конкретный sink.
Например, писать в `db_only.log` только записи, где `extra["topic"] == "db"`.

Фильтр может быть функцией:

```python
def only_db(record):
    return record["extra"].get("topic") == "db"
```

И затем:

```python
logger.add("logs/db_only.log", filter=only_db)
```

---

### 11. Исключения: `logger.exception()` и `@logger.catch`

Если вы внутри `except`, удобно писать:

```python
try:
    1 / 0
except ZeroDivisionError:
    logger.exception("Что-то пошло не так")
```

Loguru автоматически добавит текущий exception и выведет трейсбек.

`@logger.catch` ловит исключения внутри функции и логирует их:

```python
@logger.catch(reraise=False)
def will_fail():
    {}["missing_key"]
```

- `reraise=False` означает: не пробрасывать ошибку дальше (демо продолжится).

---

### 12. `patch()`: автоматическое добавление/изменение полей записи

`patch()` позволяет изменить структуру записи на лету, например добавить `extra["app"]`:

```python
def add_app(record):
    record["extra"]["app"] = "loguru-guide"

patched = logger.patch(add_app)
patched.info("Теперь extra.app добавится автоматически")
```

---

### 13. `disable()` / `enable()`: отключение логов для модуля

Можно отключить логи для конкретного “имени” (обычно имя модуля):

```python
module_name = __name__
logger.disable(module_name)
logger.info("Это не выведется")
logger.enable(module_name)
logger.info("А это снова выведется")
```

Это полезно, когда какой-то модуль слишком “шумный”.

---

### 14. Файлы: rotation / retention / compression / serialize

Одна из сильных сторон Loguru — файловые sink’и “из коробки”:

- `rotation="25 KB"` — ротация по размеру (или по времени);
- `retention=5` — хранить ограниченное число файлов/период;
- `compression="zip"` — архивировать старые логи;
- `serialize=True` — писать JSON (обычно JSON Lines: одна запись на строку).

В проекте это показано в `configure_files()`: одновременно есть текстовый лог и JSON‑лог.

### 15. Практическая рекомендация по этому проекту

В `guide_loguru.py` специально есть ограничение: **импортируется только `loguru`**.
Это сделано для учебного задания, чтобы сосредоточиться на Loguru, а не на вспомогательных модулях.
В реальных проектах вы часто будете использовать дополнительные библиотеки (например, для красивого вывода или чтения конфигов), но принцип работы Loguru остаётся тем же.
